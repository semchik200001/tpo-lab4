# 📝 Шпаргалка по лабе №4 — Нагрузочное и стресс-тестирование (JMeter)

## О чём вообще проект

Это **ЛР №4 по «Тестированию ПО» (ИТМО)**. Суть: с помощью **Apache JMeter** прогнать
веб-приложение под нагрузкой и проверить, выдержит ли оно требования по скорости ответа.
Своего «кода» тут почти нет — есть **планы тестов** (`.jmx`), **результаты прогонов**
(`.csv` + HTML-дашборды) и **Python-скрипт** для построения графиков.

### Задача по варианту
- 3 конфигурации «железа» с ценами: №1 — $3900, №2 — $5800, №3 — $6900.
- Макс. пользователей: **14**, каждый шлёт **20 запросов/мин**.
- **Порог времени отклика: 740 мс.**
- Надо: выбрать **самую дешёвую** конфигурацию, которая укладывается в 740 мс,
  а потом понять, **при какой нагрузке она ломается**.

## Три вида тестов (три папки)

| Папка | Что это | Параметры | Цель |
|-------|---------|-----------|------|
| **`load/`** | **Нагрузочное** | 3 группы по 14 потоков × 120 циклов, ramp 60с, порт 8078 | Сравнить 3 конфига, выбрать дешёвый и рабочий |
| **`stress/`** | **Стресс** | до 300 потоков, ramp 180с, loops=−1 (бесконечно), порт 8081 | Найти точку, где система перестаёт укладываться в 740 мс |
| **`load-extra/`** | **Доп. задание (Chaos)** | 70 потоков, порт 8081 | Поломать систему намеренно (рестарт сервера, сетевая задержка) |

### Поочерёдно или параллельно гоняются тесты?
Управляет флаг **`serialize_threadgroups`** в Test Plan:
- `false` или флаг отсутствует → все Thread Group стартуют **ПАРАЛЛЕЛЬНО (одновременно)**.
- `true` → поочерёдно, одна за другой.

В этом проекте:
- **`load/`** — 3 группы (Config #1/#2/#3 users), флага НЕТ → **по умолчанию параллельно**.
  То есть все 3 конфига долбятся одновременно (3×14 = 42 потока разом).
  ⚠️ На защите учти: формально это конкуренция за ресурсы машины-генератора. Но т.к. у каждого
  конфига свой URL (config=1/2/3) и свой сервер на стороне кафедры — на результаты, скорее всего,
  не влияет.
- **`stress/`** — `serialize_threadgroups=false`, но там всего ОДНА группа → вопрос неактуален.
- **`load-extra/`** — одна группа → неактуально.

**Короткий ответ:** в нагрузочном тесте 3 конфигурации гоняются **параллельно**.
В стрессе и доп. задании — по одной группе.

### Ключевые элементы в JMX-плане
- **Thread Group** — виртуальные пользователи: `num_threads` (кол-во), `ramp_time` (разгон),
  `loops` (повторы; −1 = бесконечно).
- **HTTP Request** — сам запрос к серверу.
- **Duration Assertion** — проверяет, уложился ли ответ в порог (740 мс).
- **Constant Throughput Timer** — держит нагрузку 20 запр/мин на пользователя.
- **View Results in Table / Graph Results** — визуализация.

**`load-extra/delay.py`** — mitmproxy-скрипт, добавляет **0.3с задержки** на каждый запрос
(имитация «тормозящей сети» для chaos-теста). Поэтому две папки результатов:
`result/` (без задержки) и `result-delay/` (с задержкой).

- **Конфиг №1** ($3900): среднее ~1072 мс, 100% запросов мимо порога → ❌

- Рекомендация: эксплуатировать с запасом до **70 потоков**.

## Что за графики в отчёте — разбор каждого

### 1. Графики из HTML-дашборда JMeter (`load/report*/`)
Генерятся автоматом командой `jmeter ... -e -o ./load/report`.

- **Response Times Over Time** — время отклика во времени. Видно, **растёт ли задержка**
  по мере набора нагрузки (ramp-up). Влияет на вывод «справляется/нет».
- **Throughput (пропускная способность)** — сколько запросов/сек сервер реально переваривает.
  Если throughput упёрся в потолок, а запросов всё больше → очередь, рост latency.
- **Latency vs Response time** — latency = время до **первого байта** ответа;
  response time = до **полного** ответа. Разница показывает: тормозит сеть или обработка.
- **Active Threads Over Time** — сколько виртуальных юзеров активно в каждый момент
  (показывает корректность ramp-up).
- **Таблица статистики** (`statistics.json`): `meanResTime` (среднее), `medianResTime`,
  `pct1/2/3ResTime` (перцентили 90/95/99), `errorPct` (% ошибок), `throughput`.

**Зачем перцентили?** Среднее обманчиво. **P90 = 90% запросов быстрее этого значения.**
P99 ловит «хвост» — редкие, но жуткие тормоза. Если P95 < 740 мс — большинство юзеров довольны.

### 2. График из Python (`stress/response_time_analysis.py` → `response_time_vs_load.png`)
Главный график стресс-теста. Две панели:

- **Левая — «Среднее и медианное время отклика»:**
  - Синяя линия — среднее, зелёная — медиана; по X — число потоков (нагрузка), по Y — мс.
  - **Красная пунктирная линия = порог 740 мс.**
  - Красная заливка — зона, где среднее **превысило порог** (тут система «сломалась»).
  - Скрипт сам печатает: при скольких потоках пробит порог и макс. пропускную способность.

- **Правая — «Процентили (P90/P95/P99)»:**
  - Показывает поведение не только «середины», но и **худших случаев** под ростом нагрузки.
  - P99 пробивает порог раньше среднего — «хвост» страдает первым.

> ⚠️ Нюанс: в `stress/response_time_analysis.py` (строка 5) путь к CSV **захардкожен**
> как `/Users/admin/tpo-4/...` — это чужой путь. Чтобы перестроить график у себя, поменяй
> на свой (`load-extra/result-delay/results.csv` или `stress/result/results.csv`).

## Как запускалось (для защиты)

```bash

# Доступ к серверу кафедры — проброс порта по SSH
ssh -f -N -L 8079:stload.se.ifmo.ru:8080 s409067@se.ifmo.ru -p 2222

# Графики стресс-теста
python3 stress/response_time_analysis.py
```
