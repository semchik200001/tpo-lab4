# 📝 Шпаргалка по лабе №4 — Нагрузочное и стресс-тестирование (JMeter)

## О чём вообще проект

Это **ЛР №4 по «Тестированию ПО» (ИТМО)**. Суть: с помощью **Apache JMeter** прогнать
веб-приложение под нагрузкой и проверить, выдержит ли оно требования по скорости ответа.
Своего «кода» тут почти нет — есть **планы тестов** (`.jmx`), **результаты прогонов**
(`.csv` + HTML-дашборды) и **Python-скрипт** для построения графиков.

### Задача по варианту
- 3 конфигурации «железа» с ценами: №1 — $5500, №2 — $8800, №3 — $11100.
- Макс. пользователей: **7**, каждый шлёт **20 запросов/мин**.
- **Порог времени отклика: 680 мс.**
- Надо: выбрать **самую дешёвую** конфигурацию, которая укладывается в 680 мс,
  а потом понять, **при какой нагрузке она ломается**.

## Три вида тестов (три папки)

| Папка | Что это | Параметры | Цель |
|-------|---------|-----------|------|
| **`load/`** | **Нагрузочное** | 3 группы по 7 потоков × 120 циклов, ramp 60с, порт 8078 | Сравнить 3 конфига, выбрать дешёвый и рабочий |
| **`stress/`** | **Стресс** | до 300 потоков, ramp 180с, loops=−1 (бесконечно), порт 8081 | Найти точку, где система перестаёт укладываться в 680 мс |
| **`load-extra/`** | **Доп. задание (Chaos)** | 70 потоков, порт 8081 | Поломать систему намеренно (рестарт сервера, сетевая задержка) |

### Поочерёдно или параллельно гоняются тесты?
Управляет флаг **`serialize_threadgroups`** в Test Plan:
- `false` или флаг отсутствует → все Thread Group стартуют **ПАРАЛЛЕЛЬНО (одновременно)**.
- `true` → поочерёдно, одна за другой.

В этом проекте:
- **`load/`** — 3 группы (Config #1/#2/#3 users), флага НЕТ → **по умолчанию параллельно**.
  То есть все 3 конфига долбятся одновременно (3×7 = 21 поток разом).
  ⚠️ На защите учти: формально это конкуренция за ресурсы машины-генератора. Но т.к. у каждого
  конфига свой URL (config=1/2/3) и свой сервер на стороне кафедры — на результаты, скорее всего,
  не влияет.
- **`stress/`** — `serialize_threadgroups=false`, но там всего ОДНА группа → вопрос неактуален.
- **`load-extra/`** — одна группа → неактуально.

**Короткий ответ:** в нагрузочном тесте 3 конфигурации гоняются **параллельно**.
В стрессе и доп. задании — по одной группе.

### Ключевые элементы в JMX-плане
- **Thread Group** — виртуальные пользователи: `num_threads` (кол-во), `ramp_time` (разгон),
  `loops` (повторы; −1 = бесконечно).
- **HTTP Request** — сам запрос к серверу.
- **Duration Assertion** — проверяет, уложился ли ответ в порог (680 мс).
- **Constant Throughput Timer** — держит нагрузку 20 запр/мин на пользователя.
- **View Results in Table / Graph Results** — визуализация.

**`load-extra/delay.py`** — mitmproxy-скрипт, добавляет **0.3с задержки** на каждый запрос
(имитация «тормозящей сети» для chaos-теста). Поэтому две папки результатов:
`result/` (без задержки) и `result-delay/` (с задержкой).

## Результаты (РЕАЛЬНЫЙ прогон через JMeter, 7 юзеров, порог 680 мс)

Тесты прогнаны по очереди (serialize) через SSH-туннель до сервера кафедры.

**Нагрузочное (840 запросов на конфиг):**
- **Конфиг №1** ($5500): среднее **1532** мс, медиана 1497, 100% мимо → ❌
- **Конфиг №2** ($8800): среднее **1124** мс, медиана 1098, 100% мимо → ❌
- **Конфиг №3** ($11100): среднее **743** мс, медиана **703**, min **659**, 89.4% мимо → ✅ **выбран** (кардинально лучше остальных, на грани порога)

> Парадокс варианта: самый «рабочий» — самый дорогой. Дешёвые не тянут нагрузку.
> ⚠️ Туннель добавляет сетевую задержку (~базовые 950 мс), поэтому абсолютные
> значения завышены — config #3 на грани 680 мс именно из-за этого.

**Стресс (конфиг №3, до 300 потоков):**
- Базовое время отклика на низкой нагрузке: ~950–1000 мс (уже выше порога из-за туннеля).
- При ~100 потоках: медиана ~3000–4000 мс.
- При 300 потоках: ~10000–11000 мс.
- Вывод: время отклика монотонно растёт с нагрузкой, запас минимален.

## Что за графики в отчёте — разбор каждого

Все графики строятся Python-скриптами по реальным CSV-результатам прогона. Идея — **все
метрики привязаны к количеству одновременных пользователей** (нагрузке), а не ко времени:
пользователи одинаковые, поэтому именно число юзеров — главная переменная.

### Нагрузочное тестирование (раздел «Графики пропускной способности»)

**1. Таблица статистики** (`load_stats_table.png`, скрипт `load/load_charts.py`) —
итоговые цифры по 3 конфигам: среднее, медиана, P90/P95/P99, min/max, % превышений 680 мс.
Конфиг #3 подсвечен зелёным (выбран), #1/#2 красным (не прошли).

**2. Сравнение конфигураций** (`config_comparison.png`, `load/load_charts.py`) — 3 столбчатые
панели: время отклика по конфигам (среднее/медиана/P95/P99 + порог), доля запросов > 680 мс,
пропускная способность. Наглядно видно: #3 кардинально лучше.

**3. Box-plot «с усами» по числу пользователей** (`boxplot_load.png`, `load/boxplot_load.py`) —
**главный график.** По оси X — точное число одновременных пользователей (1…7), снизу `n=` число
замеров. Для каждого числа юзеров — «ящик с усами»: коробка = 25–75 перцентиль, линия внутри =
медиана, усы = разброс, точки = выбросы. Поверх — линии **среднего (синие ромбы)** и
**медианы (зелёные квадраты)** + порог 680 мс. Видно, как с ростом юзеров растёт разброс.

### Стресс-тестирование (раздел «скрипт на Python»)

**4. Box-plot по диапазонам нагрузки** (`boxplot_stress.png`, `stress/boxplot_stress.py`) —
ящики с усами по диапазонам юзеров (1–25, 26–50, … 276–300). Снизу `n=` число замеров.
Показывает, как распределение времени отклика «уезжает» вверх с ростом нагрузки.

**5. Линия среднее + медиана от числа пользователей** (`response_time_vs_load.png`,
`stress/response_time_analysis.py`) — две линии (среднее и медиана) в зависимости от числа
потоков, порог 680 мс пунктиром, красная заливка зоны превышения. Время отклика монотонно
растёт от ~700 мс (низкая нагрузка) до ~10000–11000 мс при 300 пользователях.

### Зачем box-plot («с усами»)?
Среднее обманчиво — его задирают единичные выбросы. Box-plot показывает **сразу всё**:
медиану (типичное значение), межквартильный размах (где лежат 50 % запросов) и выбросы.
**P90 = 90 % запросов быстрее этого значения.** Если медиана и коробка ниже порога —
большинство пользователей довольны, даже если есть редкие медленные ответы.

> ⚠️ Все скрипты берут путь к CSV относительно расположения файла (хардкод чужого пути убран).
> Запуск: `./venv/bin/python load/boxplot_load.py` и т.д. (нужны pandas + matplotlib в venv).

## Как запускалось (для защиты)

```bash
# Нагрузочный тест + HTML-отчёт
jmeter -n -t ./load/test-plan.jmx -l ./load/result/results.csv -e -o ./load/report

# Доступ к серверу кафедры — проброс порта по SSH (логин s409163, порт 8078 = как в JMX)
ssh -f -N -L 8078:stload.se.ifmo.ru:8080 s409163@se.ifmo.ru -p 2222

# Графики стресс-теста
python3 stress/response_time_analysis.py
```
